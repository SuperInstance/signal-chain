The Signal Chain

Most inputs don't need a model.

How to route each stage in an AI pipeline between pure code and full model invocation — and why one dial per stage is the right abstraction.

The re-derivation problem.

Most AI pipelines re-derive the same intermediate results at every stage. A spam filter checks the sender address, then the next stage checks it again. A drift detector computes a rolling mean, then the model recomputes it from raw data. By mid-pipeline, most of the model's capacity is spent reconstructing context it already had upstream — not making new decisions.

This matters economically. A spam filter processing 100K emails per hour with a uniform full-model approach spends $3,000/hour on API costs. At 10 million emails per day, that's $4.4M per year. If 48% of those emails are resolvable by a single regex, you're spending $2.1M on model calls that return what the regex already knew.

The fix isn't "use smaller models" or "add a pre-filter." The fix is to give each processing stage its own control for how much model capacity it uses, and to carry forward every intermediate result so nothing is re-derived.

The α dial: one number per stage.

Each processing stage — called a room — has a single parameter: α ∈ [0, 1]. At α=0, pure code runs (no inference, no cost). At α=1, a full model handles everything. The pipeline is a chain of rooms, each independently tuned.

RoomαWhat runsResolvedLatency
1. Header parse0Regex + DNSBL lookup48%<0.1ms
2. Content classify0.43B model: "is this spam?"31%~10ms
3. Intent extract0.68B model: what's the email trying to do?12%~50ms

79% of emails never see a model. Rooms 1–2 handle them with regex and a small classifier. Only the hardest 9% reach rooms 3–4, and when they do, the model doesn't re-parse headers or re-classify content — it reads the tile chain and handles only the delta.

Each room writes its conclusions into a tile — a structured, content-addressed packet. Tiles carry forward:

Tile 1 (header)  → "sender: unknown, not in DNSBL, SPF pass, no DKIM"
Tile 2 (classify) → "spam probability: 0.62 (ambiguous), keywords: 'urgent', 'wire'"
Tile 3 (intent)   → "intent: financial_request, urgency_high, no prior relationship"
Tile 4 (escalate) → MODEL INVOKED — reads tiles 1–3, decides in one pass

The escalation model reads three tiles and answers one question: given an ambiguous sender, borderline spam score, and high-urgency financial request with no prior relationship — what's the call? It doesn't re-derive anything. Tiles carry the accumulated context.

Where this applies.

Any pipeline where most inputs are decidable by simple rules and only a minority need deep reasoning:

Content moderationKeyword blocklist + regex patternsContext-dependent toxicity assessment60% code, 40% model

The pattern is the same: cheap computation handles the common cases, expensive computation handles the rest, and tiles carry context so the expensive computation doesn't re-derive what the cheap computation already figured out.

Why small models work per-room.

A model scoped to one room's decision space compresses dramatically better than a general-purpose model. For binary drift detection, the decision boundary lives on a low-dimensional manifold. SplineLinear reparameterizes the weight matrix using Eisenstein lattice basis functions — 820 control points instead of 16,384 independent values:

ModelParametersAccuracyInference
Dense baseline16,384100%0.8ms
SplineLinear820100%0.3ms
LoRA adapter4,09687%0.6ms

Same accuracy, 20× fewer parameters, sub-millisecond on every CPU target. On an ESP32-S3: 4.2 KB flash, 1.8 KB RAM, 0.3ms inference. This only works because the model is solving one narrow decision — not general intelligence.

Deadband: when the model wakes up.

Deadband is the mechanism that turns α up when code can't handle the situation. Four KPIs, each with a threshold and a duration gate:

KPIThresholdDuration gate
Task completion rate< 90%5 min sustained
Average wait time> 30s30s sustained
Energy over baseline> 10%30s sustained
Inference MAE> 10%3 eval windows

A 1-second spike doesn't wake the model. Sustained degradation does. Severity ramps as severity = breach_fraction × duration_factor, where duration_factor goes from 0.3 to 1.0 over 10 minutes.

Hysteresis prevents oscillation: exiting deadband requires recovering past the threshold by 1.1×. Threshold is 90% → must hit 99% before the model goes back to sleep.

Frozen Context Windows and seeds.

When deadband triggers, the room's state is captured as a Frozen Context Window — immutable, content-addressed by SHA-256. The model receives the exact room state at the moment of failure: every tile, every KPI, every confidence score. Not a summary — the actual state, frozen, reproducible, diffable.

When a model resolves a deadband event and the fix passes validation, the response becomes a seed — a locked pattern that runs as code on future matching inputs. One model call becomes a permanent pattern match.

UNLOCKED → CANDIDATE → VALIDATING → LOCK_PENDING → LOCKED
                                                      ↓
                                              DEPRECATED → ARCHIVED

(any pre-lock stage → DISCARDED)

No reinforcement learning. No reward function — binary pass/fail on KPI recovery. No gradient updates — seeds are append-only, so catastrophic forgetting is structurally impossible. The cost of learning is one model call per deadband event.

Open problems.

Auto-tuning α across a 10-room chain is a multi-armed bandit problem with 1010 configurations. Cascading deadband — every room triggering simultaneously — could spike costs exponentially. Tiles bloat on long chains until downstream models waste their context window on history instead of signal. Stale seeds produce wrong answers when input distributions shift.

The current implementation uses a mock model backend. Production deployment requires real model adapters with batching, retries, rate limiting, and caching. The benchmark results use deterministic mock responses — real model behavior will differ.

Implementation.

The reference implementation is spreader-tool — 310 tests, zero external dependencies, pure Python. The benchmark is a single runnable file.
