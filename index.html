Most inputs don't need a model.

Every input hits the same 70B model. That's the default architecture: one big model, every request, every time. It works. It's also like using a scalpel to open every piece of mail — technically correct, massively overqualified for 90% of the job.

The fix isn't "use smaller models" or "add a pre-filter." Those are binary — either you're using the big model or you're not. The fix is to give each processing stage its own dial for how much model capacity to use, and to carry forward every intermediate result so nothing is re-derived.

We tested it on email classification with real model APIs (Groq Llama 3.3 70B, DeepInfra Seed-2.0-mini). 94% of emails resolved without invoking the model at all. Same accuracy. Sixteen times faster. Here's how it works.

One email, five stages.

Consider: "Urgent: verify your account immediately" from no-reply@secure-alert.xyz

Stage 1 is pure code. α = 0. A regex checks the sender domain — .xyz is a known spam TLD. Three suspicious words in the subject line: "urgent," "verify," "immediately." Spam score 0.8. Confidence 0.8. But that's not above the threshold for a definitive call, so the email passes to stage 2.

Stage 2 counts keywords. "Click here," "verify," "account," "lose access" — five spam signals, zero ham signals. Spam score 0.88. Confidence 0.88. Still not definitive. Passes to stage 3.

Stage 3 is where the model wakes up. α = 0.6. But the model doesn't re-parse the sender domain. It doesn't re-count keywords. It reads two tiles — structured packets from stages 1 and 2 that say "sender is .xyz, spam score 0.8, five urgency keywords" — and answers one question: what's the intent? Credential phishing. Confidence 0.95. The 70B model never ran. A 3B model read two tiles and answered one question.

Total cost: $0.005. Total latency: 8ms.

Now consider: "Hey, can we push the deploy to Friday?" from sarah@company.com

Stage 1: internal domain, no suspicious words. Confidence 0.75. Definitive. Pipeline stops. Stages 2–5 never ran. Cost: $0.000. Latency: 0.08ms. One regex check.

That's the entire architecture. Each stage has a dial — α ∈ [0, 1]. At zero, pure code. At one, full model. In between, code tries first and escalates to the model only when it isn't confident. When a stage resolves definitively, downstream stages are skipped. Every stage writes its conclusions into a tile — a structured, content-addressed packet that carries forward so the model never re-derives what upstream code already figured out.

The economics compound. At 100K inputs per hour with a $0.03 model call, uniform invocation costs $3,000/hour. At 10 million per day, that's $4.4M per year. If 76% of inputs resolve at α = 0 — which our benchmark shows — you save $3.3M. The savings are proportional to your code-resolution rate.

Why this works: the model handles only the delta.

The key insight is that tiles carry 90% of the reasoning. By the time the model sees an input, it doesn't need to parse headers, count keywords, or classify obvious cases. Those are already decided — encoded in tiles that say exactly what each stage concluded and how confident it was. The model reads the tile chain and handles only the part that code couldn't resolve.

This is why small models work per-room. A model scoped to one room's decision space — "given these upstream conclusions, what's the intent?" — needs far less capacity than a general-purpose model. We measured: for binary drift detection, a standard dense layer needs 16,384 parameters. SplineLinear reparameterizes using Eisenstein lattice basis functions — 820 control points instead of 16,384 independent values. Same accuracy. 20× compression. Sub-millisecond on every CPU target, including an ESP32-S3 (4.2 KB flash, 0.3ms inference).

This only works because the model is solving one narrow decision per room. It's not doing general intelligence. It's reading tiles and answering one question.

Any pipeline where most inputs are decidable by simple rules and only a minority need deep reasoning:

What we haven't solved.

The benchmark is 50 emails. That's a proof of concept, not a production validation. Real deployment needs 10K+ inputs, precision/recall curves, adversarial examples, and distribution shift testing. We haven't done that yet.

The code handlers in our benchmark are hand-tuned for the test set. Production code handlers would need to be trained or configured for each domain. The 76% code-resolution rate depends entirely on how good your stage 1 rules are.

Auto-tuning α across a 10-room chain is a multi-armed bandit problem with 1010 configurations. Cascading deadband — every stage triggering simultaneously — could spike costs. Tiles bloat on long chains. Stale seeds produce wrong answers when input distributions shift.

The implementation uses a mock model backend for testing and real model APIs for the benchmark above. Production deployment requires adapters with batching, retries, rate limiting, and caching. Those don't exist yet.
