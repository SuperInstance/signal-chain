Most inputs don't need a model.

Here's something we noticed: in any AI pipeline, the same expensive model gets called on every input — even the ones a regex could handle. Not because anyone thinks that's optimal. Because there hasn't been a clean abstraction for "use the model when code isn't enough."

We built that abstraction. Each processing stage gets a dial — α ∈ [0, 1] — controlling how much model capacity it uses. Most stages sit near zero and resolve inputs with pure code. The model only wakes up when confidence drops below threshold, and when it does, it reads what the code already decided instead of re-deriving everything from scratch.

The result: 94% of emails in our benchmark resolved without invoking the model at all. Same accuracy. Sixteen times faster. Tested against real APIs — Groq Llama 3.3 70B and DeepInfra Seed-2.0-mini — not mocks.

Why a dial per stage, not a pre-filter.

A pre-filter is binary: you either route to the model or you don't. But inputs aren't binary. Some need a regex. Some need a keyword counter. Some need a 3B model with a little context. Some genuinely need a 70B model. Each of these tiers has a different cost — from zero to $0.03 per call — and a binary split treats all non-obvious cases as equally expensive.

The dial makes the trade-off explicit and tunable per stage. A header-parsing stage runs at α = 0 because regex handles it. An escalation stage runs at α = 0.8 because ambiguous cases need judgment. A consistency-check stage runs at α = 0.2 because checking whether upstream rooms agree is a simple property test. Each stage gets exactly the capacity it needs — no more, no less.

And here's the part that surprised us: when a stage resolves confidently, downstream stages don't run at all. The pipeline stops. 76% of emails in our benchmark resolved at stage 1. Stages 2 through 5 never executed. No model invocation, no API call, no latency. Just a regex and a domain check.

What the model actually sees.

When the model does wake up, it doesn't re-parse headers or re-count keywords. Each stage writes its conclusions into a tile — a structured, content-addressed packet. The model reads the tile chain and handles only the delta.

An email from no-reply@secure-alert.xyz with subject "Urgent: verify your account" passes through three stages before the model is invoked. Stage 1 flags the domain and urgency words. Stage 2 counts spam signals in the body. By stage 3, the model receives two tiles that say "sender is .xyz, spam score 0.8, five urgency keywords" and answers one question: what's the intent? Credential phishing, confidence 0.95. The 70B model never ran. A 3B model read two tiles and handled the 10% that code couldn't.

Total cost: $0.005. Total latency: 8ms.

An internal email from sarah@company.com? Stage 1 sees a trusted domain, no suspicious words, confidence 0.75. Pipeline stops. $0.000, 0.08ms.

Why small models work per-room.

A model scoped to one room's decision space compresses far better than a general-purpose model, because it's solving one narrow decision — not general intelligence. For binary drift detection, a standard dense layer needs 16,384 parameters. SplineLinear reparameterizes the weight matrix using Eisenstein lattice basis functions — 820 control points instead of 16,384 values. Same accuracy, 20× compression, sub-millisecond on CPU. On an ESP32-S3: 4.2 KB flash, 0.3ms inference.

This is the deeper pattern: when you scope a model to one decision with full tile context, the decision boundary lives on a low-dimensional manifold. You don't need 70B parameters to classify intent when the tiles already tell you the sender is suspicious, the body has five urgency keywords, and the domain is .xyz. You need a small model that reads context and answers one question.

Where this applies.

Any pipeline where most inputs are decidable by simple rules and only a minority need deep reasoning. Email spam (~76% code resolution). Intent routing (~70%). Content moderation (~60%). Sensor monitoring (~90%). Document triage (~75%). The pattern is the same: cheap computation handles the common cases, tiles carry context, the model handles the delta.

What we haven't solved.

50 emails is a proof of concept, not production validation. We need 10K+ inputs, precision/recall curves, and adversarial testing. Auto-tuning α across a 10-stage chain is an unsolved optimization problem. Cascading deadband — every stage triggering simultaneously — could spike costs. Tiles bloat on long chains. Stale seeds produce wrong answers when distributions shift. The production adapters (batching, retries, rate limiting) don't exist yet.
