That's the entire architecture. Every stage has a dial for how much model capacity to use. When the code path is confident enough, downstream stages are skipped. When it's not, the model reads the tile chain and handles only the part that code couldn't.

The dial: α ∈ [0, 1] per stage.

Each processing stage has one parameter. α = 0 means pure code - regex, lookup tables, arithmetic. α = 1 means full model invocation on every input. Between, the stage tries code first and escalates to the model only when code confidence is below threshold.

αWhat actually runsLatencyCost
0Regex, rules, arithmetic, lookup tables<0.1ms$0
0.2Code first; 820-param micro-model if unsure0.3ms$0.0001
0.4Code first; 3B model on low-confidence inputs~10ms$0.002
0.6Small model by default; tiles from upstream~50ms$0.01
0.8Large model with full tile history~500ms$0.10
1.0Full model, every input, no code fallback1-10s$1.00+

The gate logic is straightforward:

def process_room(room, input_data, tile_chain):
    # Try code first
    if room.alpha < 1.0:
        code_result = room.code_handler(input_data, tile_chain)
        if code_result.confidence >= room.confidence_threshold:
            return code_result.as_tile()    # done, no model needed

    # Code wasn't confident enough - invoke model with tile context
    model_result = room.model.invoke(
        prompt = build_prompt(input_data, tile_chain),
        max_cost = room.max_cost_per_call
    )
    return model_result.as_tile()

Most rooms exit on the code path most of the time. The model handles the delta - the part the shell doesn't already know.

Tiles: structured context packets.

A tile is a content-addressed, immutable record of what one room decided. It carries the room ID, the confidence score, a label, and arbitrary metadata. Tiles accumulate through the pipeline - room 3 reads tiles from rooms 1 and 2, then emits its own.

@dataclass(frozen=True)
class Tile:
    room_name: str           # "header_parse"
    label: str | None        # "spam", "ham", "uncertain"
    confidence: float        # 0.0-1.0
    metadata: dict           # {"domain_suspicious": True, "keyword_hits": 5}
    cost: float              # $0.005
    latency_ms: float        # 8.2
    invoked_model: bool      # False for pure code

Content-addressed means two rooms that see the same input and produce the same result emit tiles with the same hash. Deduplication is automatic - if room 1 already classified this sender as spam last week, the tile is already stored.

The practical effect: when the 70B model runs in room 4, it doesn't re-parse headers, re-count keywords, or re-classify content. It reads three tiles and answers one question. The tiles carry 90% of the reasoning. The model handles the remaining 10%.

Early exit: most inputs don't reach the end.

When a room resolves with confidence ≥ threshold and its label is definitive (“ham” or “spam”), the pipeline stops. Downstream rooms are skipped entirely — no invocation, no cost, no latency.

Deadband: turning α up automatically.

The α dial doesn't have to be static. Deadband detection monitors KPIs and turns α up when code can't handle the situation:

KPIBreach thresholdDuration gateWhat happens
Task completion< 90%5 min sustainedSeverity ramps 0.3 → 1.0
Average wait time> 30s30s sustainedFCW created, model notified
Inference error (MAE)> 10%3 eval windowsSeed unlocked, model invoked

Hysteresis prevents flickering: exiting deadband requires recovering past the original threshold by a 1.1× factor. Not just back to 90% completion - up to 99%. Anyone who's been on-call at 3 AM knows why this matters.

When deadband opens, the room's state is captured as a Frozen Context Window - an immutable, copy-on-write snapshot. The model receives the exact room state at the moment of failure, not a summary. You can replay any decision, diff any two snapshots, trace exactly why the system made every choice.

Seeds: locking validated patterns.

When a model response passes validation, it gets locked as a seed. The seed captures the pattern - "for inputs like this, this response worked." Next time a similar input arrives, the code path matches the seed instead of invoking the model.

Seed lifecycle:
  UNLOCKED → CANDIDATE → VALIDATING → LOCK_PENDING → LOCKED
                                                         ↓
                                              (if KPIs degrade)
                                                         ↓
                                              ESCALATING → DEPRECATED → ARCHIVED

Locked seeds are the mechanism for distillation. Run a room at high α for a while, collect input-output pairs, identify the dominant patterns (what the model does 90% of the time), and lock those as seeds. The seeds handle the common cases. Deadband catches the rest by turning α back up temporarily.

Where this pattern applies.

Any multi-stage pipeline where most inputs are decidable by simple rules and only a minority need deep reasoning:

DomainRoom 1 (pure code, α=0)Escalation room (α=0.8)Code resolution rate
Email spamSender domain + keyword countSemantic intent analysis~48%
Intent routingExact match on common intents ("help", "stop")Ambiguous utterance classification~70%
Content moderationBlocklist + regexContext-dependent toxicity~60%
Sensor monitoringThreshold + rolling statisticsDistribution shift analysis~90%
Document triageHeader + metadataFull document understanding~75%

The economics: at 100K inputs/hour with a $0.03/model call, uniform invocation costs $3,000/hour. If 48% resolve at α=0, the signal chain costs $1,560/hour. At 10M inputs/day, that's $34M/year vs $18M/year. The savings are proportional to your code-resolution rate.

Compression: small models for narrow decisions.

A model scoped to one room's decision space compresses far better than a general-purpose model. The decision boundary for binary drift detection lives on a low-dimensional manifold - SplineLinear reparameterizes the weight matrix using Eisenstein lattice basis functions:

20× compression, same accuracy, sub-millisecond on CPU. On an ESP32-S3: 4.2 KB flash, 1.8 KB RAM, 0.3ms inference. This works because the model solves one narrow decision per room - not general intelligence.

Open problems.

Auto-tuning α across a 10-room chain is a multi-armed bandit problem with 1010 configurations. Cascading deadband - every room triggering simultaneously - could spike costs. Tiles bloat on long chains. Stale seeds produce wrong answers when input distributions shift.

The current benchmark uses a deterministic mock model backend. Production deployment requires real model adapters with batching, retries, and rate limiting.
